<action>
Create `docs/rate-limiting.md` with the following sections:

1. **Overview** — describes that rate limiting uses Flask-Limiter with in-memory storage (default `memory://`) and configurable per-endpoint limits, designed for local network deployments with small user groups.

2. **Configuration Reference** — table with the following environment variables: `RATE_LIMIT_GLOBAL` (default: `100 per minute`), `RATE_LIMIT_MODELS` (default: `30 per minute`), `RATE_LIMIT_CHAT` (default: `30 per minute`), `RATE_LIMIT_STORAGE_URL` (default: `memory://`).

3. **Per-Endpoint Limits** — documents which endpoints have specific limits: `/v1/models` uses `RATE_LIMIT_MODELS`, `/v1/chat/completions` uses `RATE_LIMIT_CHAT`, `/metrics` is exempt, `/health` and `/ready` use the global limit.

4. **Client Identification** — describes that the key function respects the `X-Forwarded-For` header for proper client identification behind proxies.

5. **Redis Configuration** — notes that setting `RATE_LIMIT_STORAGE_URL=redis://localhost:6379` enables Redis-backed distributed rate limiting for multi-worker deployments; the `redis` package must be installed.

6. **Examples** — sample `.env` configuration for a local deployment with 60 req/min global, 20 req/min for chat.
</action>

<action>
Create `docs/debugging.md` with the following sections:

1. **Overview** — describes that every request is assigned a unique request ID (8-character UUID prefix) at the start of the request lifecycle, accessible via `X-Request-ID` response header and included in all log statements.

2. **How Request IDs Work** — explains the flow: generated in `before_request` hook (or taken from incoming `X-Request-ID` header), stored in `g.request_id` and a `ContextVar`, added to all log records via `RequestIDFilter`, and returned as `X-Request-ID` response header.

3. **Tracing a Request Through Logs** — shows log format `%Y-%m-%d %H:%M:%S | LEVEL | name | [request_id] | message` with example log lines for request start, completion, and errors.

4. **Client-Side Correlation** — describes how clients can send their own `X-Request-ID` header to correlate proxy logs with client-side events.

5. **Error Response Format** — documents that all error responses include a `request_id` field in the JSON body matching the log entry, enabling cross-referencing.

6. **Body Logging** (only if plan 02-02 is completed) — documents the `LOG_REQUEST_BODY` and `LOG_RESPONSE_BODY` config options with a security warning that enabling body logging may expose sensitive data. Reference the `.env.example` security warning.

7. **Troubleshooting** — `request_id` shows as empty string `[]` in logs: confirm `init_request_id_middleware(app)` is called in `create_app()` before any routes are accessed.
</action>
